# Environment Protocol: Zero-Skew Training/Eval Interface

## Problem

Today we have **two parallel implementations of the ARC agent loop**:

1. **Eval:** `agent-runner.ts` → Orchestrator → `parseReplBlocks()` → `EvalRuntime.eval()` → `toolResultToUserMessage()` format → `submit()` + `agent.abort()`
2. **Training:** `arc_interaction.py` → `parse_tool_call()` (Python) → `SandboxSession.eval()` → bare `output or "(no output)"` format → submitted check

These diverge in **5 critical ways**:

| Aspect | Eval (agent-runner.ts) | Training (arc_interaction.py) |
|--------|----------------------|------------------------------|
| **Code parser** | `parseReplBlocks()` — regex for `` ```javascript ``/`` ```js ``/`` ```repl `` | `parse_tool_call()` — regex for `<tool_call>` XML |
| **Feedback format** | `Code executed:\n\`\`\`javascript\n{code}\n\`\`\`\n\nREPL output:\n{output}` | `output or "(no output)"` |
| **Output formatting** | `eval-tool.ts`: stdout + `→ returnValue` + `ERROR: ...` + optional status | `arc_interaction.py`: stdout, error appended |
| **Thinking extraction** | `extractThinking()` strips `<think>` blocks | Not handled |
| **Sub-agents** | Supported via `spawnAgent()` in scope | Not available |

This skew means the model sees **different feedback during training vs eval**. The prompt format, error messages, and conversation structure all differ. This is the #1 risk for training producing non-zero rewards — the model may learn behaviors that only work with training-specific feedback formatting.

## Goals

1. **Zero parsing skew** — single parser (TypeScript `parseReplBlocks`) for both modes
2. **Zero feedback format skew** — single formatter (TypeScript `eval-tool.ts` pipeline) for both modes
3. **Sub-agent support in training** — `spawnAgent()` available in training (future extension point, architecture must support it)
4. **Concurrent eval** — run N environments in parallel
5. **Framework-agnostic** — `reset()/step()/close()` works with verl, Tinker, SkyRL, or a for-loop

## Non-Goals

- We are NOT building ARES's full dm_env with queue-mediated LLM client. That pattern solves a different problem (LLM interception for agents that make direct API calls). Our agents use an eval sandbox, not bash commands.
- We are NOT building a container abstraction. Node VM is sufficient for ARC.
- We are NOT changing the eval CLI (`agent-runner.ts`). It stays as-is for standalone eval.

## JSON-Line Protocol (sandbox-agent.ts)

### Init

```
→ {"type":"init","task":{"train":[...],"test":[...]}}
← {"type":"ready","systemPrompt":"...","userMessage":"..."}
```

`systemPrompt` is generated by `generateCodeBlockSystemPrompt(adapter)`.
`userMessage` is the initial task prompt (same string agent-runner.ts passes to `orchestrator.run()`).

These let the RL framework construct the initial conversation: `[{role:"system", content:systemPrompt}, {role:"user", content:userMessage}]`.

### Step

```
→ {"type":"step","assistantText":"I'll analyze the examples...\n```javascript\nfunction transform(grid) {...}\n```"}
← {"type":"observation","feedback":"Code executed:\n```javascript\n...\n```\n\nREPL output:\n...","reward":0.0,"done":false,"info":{"turns":1,"submitted":false}}
```

Processing pipeline inside sandbox-agent.ts:
1. `extractThinking(assistantText)` → strip `<think>` blocks
2. `parseReplBlocks(cleanText)` → extract code segments
3. For each code segment: `runtime.eval(code)` → format output using eval-tool.ts logic (stdout + returnValue + error + optional status)
4. Format feedback using `toolResultToUserMessage` format
5. If multiple code blocks: execute sequentially, return feedback for the last one (with all side effects applied)
6. Check if `submit()` was called → set `done=true`, compute reward
7. If no code blocks found: return feedback `"No code found. Write a \`\`\`javascript block to execute code."`

### Close

```
→ {"type":"close"}
```

Process exits.

## Python Interface: ArcEnv

File: `rl/arc_env.py`

```python
class ArcEnv:
    """Environment wrapping sandbox-agent.ts for zero-skew training/eval.

    One subprocess per environment instance. Variables persist across steps.
    """

    def __init__(self, task: dict, timeout: float = 30.0):
        """Spawn sandbox-agent.ts, send init, receive system prompt + user message."""

    def reset(self) -> dict:
        """Return initial observation.

        Returns:
            {"systemPrompt": str, "userMessage": str}
        """

    def step(self, assistant_text: str) -> tuple[str, float, bool, dict]:
        """Execute one agent turn.

        Args:
            assistant_text: Raw model output (may contain ```javascript blocks, <think> blocks, prose)

        Returns:
            (feedback, reward, done, info)
            - feedback: Formatted string to append as next user message
            - reward: 0.0 during episode, 1.0 or 0.0 on terminal step
            - done: True if submit() was called or error
            - info: {"turns": int, "submitted": bool, ...}
        """

    def close(self):
        """Kill subprocess."""
```

## Integration Points

### Training (arc_interaction.py rewrite)

```python
class ArcInteraction:
    def __init__(self, config=None, tasks=None):
        self.envs: dict[str, ArcEnv] = {}

    async def start_interaction(self, instance_id=None, **kwargs):
        task = kwargs.get("task") or self.tasks.get(kwargs.get("task_id"))
        self.envs[instance_id] = ArcEnv(task)
        return instance_id

    async def generate_response(self, instance_id, messages, **kwargs):
        env = self.envs[instance_id]
        # Extract latest assistant content
        assistant_text = messages[-1].get("content", "") if messages[-1].get("role") == "assistant" else ""
        feedback, reward, done, info = env.step(assistant_text)
        return done, feedback, reward, info

    async def finalize_interaction(self, instance_id, **kwargs):
        env = self.envs.pop(instance_id, None)
        if env:
            env.close()
```

### Concurrent Eval (eval_loop.py enhancement)

```python
async def evaluate_tasks_concurrent(tasks, vllm_base_url, model, max_parallel=4, **kwargs):
    sem = asyncio.Semaphore(max_parallel)

    async def run_one(item):
        async with sem:
            return _run_agent(item["id"], item["task"], vllm_base_url, model, **kwargs)

    results = await asyncio.gather(*[run_one(t) for t in tasks])
    return results
```

Note: concurrent eval uses the existing `_run_agent()` subprocess approach (agent-runner.ts), not ArcEnv. ArcEnv is for training where the RL framework controls token generation. Eval still uses agent-runner.ts because it needs the full Orchestrator loop with real LLM calls.
