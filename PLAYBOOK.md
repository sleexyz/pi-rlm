# Project Playbook

## Strategies and Hard Rules

- **Token depth = accuracy for agentic solvers** — On ARC-AGI-2, 3.5x more tokens/attempt → 20pp more accuracy (594k tokens at 65.8% vs 2M tokens at 85.3%). Same model, same reasoning level. When reproducing results, match token expenditure not just configuration flags. Check turn limits, timeouts, and action budgets — these gate exploration depth.
- **Arcgentica's `success` field is self-eval, not ground truth** — Their attempt JSON `test_results[].success` reflects the agent's internal check, not whether the output matches the ARC answer key. Actual scoring requires running `score.py` against the solutions.

- **Edit existing files, Write only new ones — unless rewriting >50%** — For files that already exist, always use Read → Edit, even if the spec says to "update" them. Reserve Write for files that genuinely don't exist yet. The Write tool can error on existing files if they weren't Read first in the current session. Before creating any file, Glob for it first — prior sessions may have already created it. **Exception:** When a diff will delete/change more than half the file's lines, a single `Write` is cleaner than multiple `Edit` calls. Edits risk ordering conflicts and require careful context matching. The runs session rewrote 85/218 lines of `runner.ts` in one shot with zero retries.
- **Well-specified predictions eliminate design-time** — When the PROMPT includes concrete implementation approaches (not just "implement X" but "use Object.defineProperty with getters on a closure"), Ralph executes without deliberation. Invest time in prediction specificity. For 5+ file builds, specify every function signature (name, parameters, return type, defaults) — the arc session created 7 files in 8.5 minutes because nothing was left to deliberate.
- **Port reference implementation prompts, not just code** — When adapting an existing system (like arcgentica → pi-rlm), study the system prompts first. Battle-tested prompts encode methodology that transfers across languages. The code is secondary — the prompt is the product.
- **Check git status exactly once at session start** — Run `git rev-parse --git-dir 2>/dev/null` early. If it fails, skip all git operations for the rest of the session. Don't retry.
- **Batch typecheck+test is fine for TypeScript event-plumbing work** — When changes are purely additive (new fields on objects, new code paths guarded by flags), running `just typecheck && just test` once at the end catches everything. No need to run after each file edit.
- **Verify code state by grep, not by checkbox** — At session start, grep the actual source for key patterns (`submit\(`, `\bvar\b`, `snake_case_names`) rather than trusting `[x]` markers in the prompt file. Checkboxes from prior sessions may be stale, inverted, or wrong. Spend 30 seconds grepping; save 10+ minutes of wasted work.
- **When adding post-processing to IIFE-wrapped eval code, use `__retval` not `return`** — The eval runtime wraps code in `(async () => { ... })()`. If you need to append code after the user's statements (e.g., `__persist` calls), never use `return` for implicit returns — append `__retval = (expr)` instead, then `return __retval` at the very end. This ensures all post-processing code is reachable.
- **Stage specific files in commits, never `git add -A`** — `git add -A` in a repo with downloads/, symlinks, or prior session artifacts will capture hundreds of unrelated files. Always use `git add <specific-files>` or at minimum `git add pi-rlm/`. The arc-parity session's feature commit correctly staged specific files; the termination protocol's hardcoded `git add -A` pulled in everything.
- **Combine tasks that touch the same file** — When two tasks both modify the same file (like pass@K scoring + default model/thinking both in `arc-runner.ts`), implement them together. Avoids re-reading/re-editing and produces cleaner diffs.
- **Explore unfamiliar data with python3 one-liners before coding the transformer** — For JSONL/JSON data transformation tasks, run 3-5 python3 one-liners to: (1) count event types, (2) print one example of each type, (3) trace the join keys between files, (4) verify assumptions about field presence. This takes ~90 seconds and eliminates all guesswork from the implementation.
- **Test the full pipeline on real data, not just each stage** — For multi-stage pipelines (A→B→C), run the entire pipeline end-to-end before shipping. Stage-level validation (A→B looks correct, B→C looks correct) can miss information loss at boundaries. The trace-export session caught a missing `system_prompt` only when running session→rollout→sharegpt as a complete pipeline.
- **Translate at the API boundary, not at every consumer** — When changing a data format (e.g., event-stream → message-log), add a server-side converter that emits the old format rather than rewriting all frontend/consumer code. This preserves working code and makes the migration incremental. (Demonstrated by `convertMessagesToEvents()` saving ~300 lines of frontend rewrite.)
- **Build `dist/` before cross-package typecheck** — After modifying `pi-rlm/src/`, run `cd pi-rlm && bun tsc -p tsconfig.json` before running `cd domains/arc-agi-2 && bun tsc --noEmit`. The workspace dependency resolves via `dist/`, not source. `bun tsc --noEmit` validates types but doesn't produce output files.
- **Run-level resume (skip completed) beats session-level resume (replay conversation)** — For batch runners, `--resume <run-name>` that reads a results file and skips completed tasks is simpler, more robust, and more useful than resuming a single agent conversation mid-stream. The completed-set approach needs zero knowledge of session internals — just a results file.
- **Inline ASCII mockups in PROMPT specs for UI work** — The viewer-runs spec included an ASCII sidebar mockup (`▼ stellar-meadow  6/6  $1.75 / ├ 007bbfb7  ✓  $0.12  30s`). Ralph implemented it exactly as drawn with no clarification round-trips. For any UI-facing task, include a concrete visual mockup — it eliminates ambiguity about layout, hierarchy, and information density.
- **Create ephemeral test fixtures for server verification** — For HTTP server changes, create minimal fixture data (`run.json` + JSONL), start the server, curl the endpoints, verify responses, then clean up. This takes ~60 seconds and catches API shape mismatches that typecheck alone won't find. Pattern: `mkdir -p fixture → write data → start server → curl → verify → rm -rf fixture`.
- **Prefer dependency injection over monkeypatching for Python test mocks** — When a function accepts optional `llm=None, tokenizer=None` params with production defaults, pass mocks directly. Don't patch `module.vllm` or `module.AutoTokenizer` — DI is more explicit, survives refactors, and avoids `AttributeError` when import structure changes.
- **Verify remote data before assuming download succeeded** — After downloading from Modal volumes (or any remote storage), check file sizes locally (`wc -l`, `stat -f%z`). Better: add a server-side verify function that reads files and reports structure, isolating "data doesn't exist" from "download is broken."
- **Verify relative path calculations with `ls` at spec time** — When a spec specifies a relative path from `import.meta.dirname` (e.g., `../../viewer/` from `src/`), run `ls` from the source directory to verify before writing the spec. The live-viewer spec got the depth wrong (`../../viewer/` vs `../viewer/`), caught cheaply by Gate 1 but avoidable entirely.
- **Pre-check token count before vLLM `generate()` calls** — Count conversation tokens and compare against `max_model_len` before calling `llm.generate()`. When it would overflow, log the task as failed with a `context_overflow` reason and continue to the next task. This prevents one task from crashing an entire eval run.
- **Monitor long-running eval processes via checkpoint files, not task IDs** — For eval runs that write incremental checkpoints (`run.json`), monitor progress by periodically reading the checkpoint (`python3 -c "import json; ..."`) rather than relying on background task tracking. Task IDs can be lost across context boundaries; checkpoint files are persistent.
- **Bypass middleware, don't fork it** — When an external library's streaming path doesn't support your feature (e.g., logprobs), add a non-streaming bypass at the application layer rather than forking the library or threading through layers. Pattern: `if (needsFeature) { directFetch() } else { stream() }`. The unified-harness session bypassed `streamSimple` (from `@mariozechner/pi-ai`) with a direct `fetch()` to the OpenAI-compat endpoint when `topLogprobs > 0`, avoiding any changes to the external package.
- **When a prior session already did the work, verify don't redo** — At session start, check `git log`, `git diff --name-only HEAD`, and `Glob` for expected output files before implementing anything. Spending 2 minutes verifying saves 15+ minutes of redundant implementation.

## Code Patterns

- **`createTaggedOnEvent()` returns `{ handler, setUserMessage }`** — This is the pattern for event tagging in pi-rlm. The handler is passed to `agent.subscribe()`, and `setUserMessage()` is called lazily in `run()`/`call()` when the user message becomes known. Don't try to pass userMessage at construction time.
- **Debug vs normal TUI rendering** — `handleEvent` dispatches to `handleNormalEvent` or `handleDebugEvent`. They share only the loader component and `formatToolResultRaw`. The debug path uses plain `Text` components (not `Markdown`) to avoid streaming flicker.
- **Event depth vs spawn depth** — `eventDepth` (UI nesting level) and `currentDepth` (max-depth recursion guard) are separate fields in `AgentHandleOptions`. They increment together but serve different purposes.
- **AgentEvent has 8 types, not 6** — The actual `AgentEvent` union includes `turn_start`, `turn_end`, and `tool_execution_update` in addition to the commonly referenced 6. The debug renderer handles the important ones and silently ignores the rest via a switch default.
- **Cross-cutting concerns use separate subscribers** — Don't inline usage tracking or logging into the event-tagging handler. Subscribe separately via `agent.subscribe()`. The tagged handler's job is decoration; other subscribers consume the decorated events. This keeps responsibilities clean and is easier to test independently.
- **ARC adapter is a pure scope provider** — After the idiomatic refactor, the ARC adapter no longer extends `DomainAdapter` with `getAnswer()`. The `orchestrator.run()` return value IS the answer. The adapter just provides scope and prompts. For new domains, follow this pattern: adapter provides scope, orchestrator handles completion.
- **`let`/`const`/`var`/`function` all persist in EvalRuntime** — `scanDeclarations()` handles all declaration types. All persist across eval calls via `__persist`. The scanner uses `computeBraceDepth()` to only capture top-level declarations (skips nested function bodies).
- **`resolve(value)`/`reject(reason)` for orchestrator completion** — The orchestrator uses `resolve(value)` to complete with a result and `reject(reason)` to abort. There is no longer a separate domain-level submission function. `orchestrator.run()` returns the resolved value directly; reject throws.
- **DomainAdapter.generateSystemPrompt bypass pattern** — `adapter.generateSystemPrompt?.() ?? generateSystemPrompt(adapter)` is a one-line opt-in for adapters to bypass orchestrator framing. No mode flags, no conditional logic. Other adapters get the default orchestrator prompt unchanged. Use this for any "domain X needs special handling" without polluting the generic path.
- **Shared mutable counter for flat resource pools** — `{ count: number }` object passed by reference through all `createSpawnAgent` levels gives a flat pool counter in ~10 lines. Works because JS objects are pass-by-reference. Simpler than depth tracking for "total N agents" limits (vs "N levels deep" limits).
- **SessionLogger `message_end` with `usage` = assistant turn** — The session logger echoes both assistant messages and user/tool-result messages as `message_end` events. The discriminator: assistant turns have a `usage` field (input/output token counts); echo messages don't. This is the key for reconstructing conversations from JSONL.
- **Content blocks use `toolCall` not `tool_use`** — pi-agent-core serializes tool calls as `{type: "toolCall", arguments: {code}}`, not the Anthropic API's `{type: "tool_use"}`. The trace converter handles both for robustness.
- **Orchestrator root agent needs separate snapshot** — The Orchestrator creates the root `Agent` directly (not via `AgentHandle`), so cross-cutting hooks like message snapshotting must be added in both `AgentHandle.call()` and `Orchestrator.run()`. When adding behavior that should apply to "every agent," check both code paths.
- **`SessionDir` + `AgentLogger` for persistent per-agent logs** — `SessionDir` manages `logs/{domain}/{session_id}/` with auto-incrementing agent indices. `AgentLogger` writes per-agent JSONL (`agent-N.jsonl`) with `{type:"session"}` header, `{type:"message"}` entries, and `{type:"session_end"}` footer. The old `SessionLogger` (event-stream) is kept only for `cli.ts` and `trace-export.ts`. **Custom nesting:** Pass `logDir: runs/<name>/sessions` and `sessionId: "<taskId>_a<attempt>"` to nest session logs inside a run directory. No need to hack SessionDir internals.
- **`run.json` as incremental checkpoint** — Write full JSON (config + results so far) after each task completes. JSON doesn't support append, but the file stays small (few KB for 120 tasks). On resume, read `run.json`, build a `completedTaskIds` set, skip those tasks. Survives interruption at any point.
- **Composite IDs for nested filesystem resources** — When resources are nested (runs → sessions), use `<parent>/<child>` as the canonical ID everywhere: API paths, WebSocket subscriptions, DOM data attributes, URL parameters. This maps trivially to filesystem paths (`path.join(runsDir, id)`) and avoids ambiguity when the same child name exists under different parents.
- **Three-level directory watching pattern** — For nested directory structures (`runs/` → `runs/<name>/sessions/` → session dirs), create watchers at each level where the parent watcher spawns child watchers on directory creation. Store child watchers in a Map keyed by directory name for cleanup. Each level handles only its own events (new subdirectory vs new files).
- **`@modal.web_server` + subprocess + volume reload loop** — For deploying non-Python servers (Bun, Node) on Modal: (1) `@modal.web_server(port=N)` decorator, (2) `subprocess.Popen` to start the server, (3) `threading.Thread` with `volume.reload()` loop so the server sees cross-container writes. The Python wrapper handles volume sync while the subprocess handles HTTP/WebSocket.
- **Transient server-computed fields use underscore prefix** — When the API response needs data that isn't in the persisted JSON (e.g., `_activeSessions` on `run.json` response), prefix with `_` to signal "server-added, not in the file." Consumers can rely on the field existing in API responses but know not to expect it in the raw JSON file.
- **`onPayload` callback for provider-specific params in pi-ai** — `streamOpenAICompletions(url, messages, options)` calls `options.onPayload?.(requestBody)` before sending, allowing mutation of the raw OpenAI request. Use this to inject `top_p`, `frequency_penalty`, or any provider-specific params without modifying the library. Example: `onPayload: (p) => { p.top_p = 0.95 }`.
- **Non-streaming fetch for logprobs in createCodeBlockStreamFn** — When `topLogprobs > 0`, `createCodeBlockStreamFn` sends a direct `fetch()` to `/chat/completions` with `stream: false, logprobs: true, top_logprobs: N`. The response `choices[0].logprobs.content` is attached to the assistant message as `(msg as any).logprobs`. This bypasses the streaming path entirely and is the only way to get per-token logprobs through the pi-ai middleware.

## Pitfalls

- **Use `git add <specific-files>` for commits** — Never `git add -A` since the workspace contains large downloads/ and other content excluded by `.gitignore`.
- **Spec says "Create file X" but it may already exist** — Always verify before writing. Existing implementations may have design improvements over the spec.
- **Manual testing requires a live API key** — `pi-rlm --debug` hits a real LLM API. There's no mock adapter for offline testing of the TUI rendering pipeline. Consider adding a `--dry-run` or replay mode.
- **Multiple `agent.subscribe()` calls are additive** — pi-agent-core allows multiple subscribers on the same agent. They receive the same event object reference. If subscriber A mutates the event (adding `_agentId`, `_depth`), subscriber B sees those mutations. Useful for cross-cutting concerns but subscriber ordering matters.
- **CLI flag parsing without a library is brittle** — Optional-argument flags like `--log [dir]` require index-based `process.argv` filtering to distinguish `--log ./logs` from `--log --debug`. Works for 2-3 flags but will break if more are added. If the flag count grows past 4, adopt `commander` or `yargs`.
- **Export `resetCounter()` for any module-level state** — Module-level counters (like `nextAgentId` in event-tagging) persist across test files. Without a reset function, test assertions on specific IDs become order-dependent and flaky. Always export a reset function alongside any module-level mutable state.
- **Use `process.cwd()` for test fixture paths in vitest** — Under symlinked repos, `__dirname` and `import.meta.url` resolve through the symlink to the real filesystem path, breaking relative paths to fixtures. `process.cwd()` works because vitest sets cwd to the project root (`pi-rlm/`). This is vitest-specific.
- **Reflector overhead can exceed 50% of core session cost** — For fast sessions (<10 min), the termination + reflector + curator pipeline ran 2-3 times redundantly in the arc session, costing ~$1.86 on top of a $3.11 core session. Consider gating reflector runs on session duration or cost threshold (e.g., skip if core session < $2).
- **Reflector agents may edit source code during analysis** — The arc session's third reflector run modified `arc-runner.ts` while reading the log. Constrain reflector agents to read-only tools + the delta file write. Surface source fixes as delta recommendations, don't apply directly.
- **`scanDeclarations` is scope-blind — use `computeBraceDepth()`** — The declaration scanner matches `let`/`const`/`var`/`function` with a regex. It will find declarations inside nested function bodies, arrow functions, and class methods. Always use `computeBraceDepth()` to filter to depth-0 declarations only. The old `var` scanner had this same bug but it was never caught because no tests used `var` inside function expressions.
- **Framework-level API renames touch more files than you expect** — When renaming something like `submit()` → `resolve()`, grep all of `src/` after the obvious renames. In the idiomatic session, `memories.ts`, `demo-adapter.ts`, and `agent-handle.ts` all had straggler references that would have been missed without a catch-all grep.
- **Predict signal-to-noise, not raw size** — When predicting whether a refactored prompt/config will be "shorter," the real metric is "no irrelevant content," not "fewer characters." A domain-specific prompt can be longer than a generic one and still be better because every byte is task-relevant. Frame predictions around content relevance, not length.
- **Keyword-matching for banned prompt content needs context** — ARC_PREMISE uses "delegate" in "do not delegate unless genuinely helpful" — this is solver-appropriate language telling the agent NOT to delegate, not orchestrator framing. Checking prompts for banned words requires understanding context, not just substring matching.
- **Intermediate formats must carry downstream metadata** — When format X feeds into format Y, X must carry everything Y needs — not just what X's own spec requires. The sparse-v1.0 Rollout spec doesn't include `system_prompt` at the top level (it's in Steps), but the ShareGPT exporter needs it. Check data flow through the full pipeline before finalizing schemas.
- **Termination protocol `git add -A` conflicts with playbook** — The hardcoded `git add -A && git commit` in the termination template conflicts with the "stage specific files" rule. When following a template that says `git add -A`, override it with specific file staging. The template instruction is generic; the playbook rule is project-specific and takes priority.
- **Spec removal tasks need consumer grep first** — Before writing a "remove X" task in a PROMPT spec, run `grep -r 'import.*X\|from.*X' --include='*.py' --include='*.ts'` to enumerate all consumers. Violated twice: session-logs spec missed `cli.ts`/`trace-export.ts` as `SessionLogger` dependents; unified-harness spec said "delete parser.py/js_sandbox.py" but `arc_interaction.py` still imports both for training. If any consumer can't be updated in the same session, change "delete" to "keep (still needed by Y)".
- **Reflector must not read its own live log** — `ralph-{session}.log` is appended to during the reflector step. Reading it creates a recursive growth loop (reflector reads → log grows → reflector reads again). Either snapshot the log before the reflector runs, or pass a read-only copy. The runs session's reflector produced no output despite ~5 read attempts.
- **PROMPT files may be gitignored — check `git status` before staging** — The termination protocol's commit step may try to stage `PROMPT-*.md` files. If the file doesn't appear in `git status` output, it's gitignored — skip it. Don't blindly include it in `git add`.
- **Capture server PIDs explicitly, don't rely on job control** — When starting a background server for testing, use `bun server.ts &; SERVER_PID=$!` and `kill $SERVER_PID` rather than `kill %1`. Job control (`%1`) is fragile across multiple sequential server starts in the same shell. If the first kill fails silently, the port stays bound and subsequent starts fail.
- **`bun install` with `workspace:*` deps fails outside the monorepo** — When deploying a single package from a bun/pnpm workspace to a container, `package.json` with `"dep": "workspace:*"` will fail to resolve. Workaround: `bun add <specific-packages>` directly in the image build, bypassing package.json entirely.
- **Periodic rescan of directories needs an "already ended" set** — When converting a one-shot directory scan into a periodic loop, items that were processed and cleaned up will be rediscovered on the next scan. Maintain a `Set` of ended/completed IDs and check it before re-processing. Without this, the rescan creates an infinite rediscovery loop.
- **Modal API renames (2025): `keep_warm` -> `min_containers`, `allow_concurrent_inputs` -> `@modal.concurrent`** — Old names still work but produce deprecation warnings. Use `min_containers=1` and `@modal.concurrent(max_inputs=100)` decorator for the current API.
- **`modal volume get` silently produces 0-byte files on 404** — When blob storage returns 404 for deeply nested paths (e.g., `sessions/*/agent-0.jsonl`), the command exits 0 and creates empty local files. Always verify sizes after download. Workaround: tar files server-side with `pack_run`, then download the single tar.
- **Modal CLI auto-generates flags — no `--` separator** — `modal run app.py::func --param value`, NOT `modal run app.py::func -- --param value`. The `--` separator silently ignores all arguments.
- **vLLM v0.11 changed `generate()` to dict-style prompts** — `llm.generate([{"prompt_token_ids": ids}], sampling_params)`, not the old `prompt_token_ids=` kwarg. When specs reference external library APIs, note the target version.
- **Claude runner (bun) can die silently mid-run** — The bun runner process has been observed dying after completing partial tasks with zero error output. Always use `--resume <run-name>` to pick up where it left off. For production baselines, consider a wrapper script that polls process liveness and auto-resumes.
- **Background task IDs in Claude Code are fragile across multiple launches** — When you launch multiple background tasks, checking `TaskOutput` for earlier tasks may fail silently. Fall back to `ps aux | grep` and reading log files (`tee -a bashes.log`). Don't trust that `TaskOutput` will always find your process.
- **Workspace `dist/` must be in Modal container images** — When deploying a bun workspace to Modal where package A depends on package B via `"B": "workspace:*"` and B's `package.json` has `main: "./dist/index.js"`, B's `dist/` directory must be copied into the image. `bun install` creates the workspace symlink, but resolution follows `main` to `dist/`. Missing `dist/` causes silent import failures at runtime, not build time.

## Environment

- **Project structure** — Source is at `pi-rlm/src/`. Tests via `just test`, types via `just typecheck`.
- **pi-tui Text component** — `Text(text, paddingX, paddingY)` with `setText()` for updates. Use for streaming content in debug mode. `Markdown` component exists but causes flicker on rapid updates.
- **ARC-AGI-2 data** — `downloads/ARC-AGI-2/data/training/` (1000 tasks), `downloads/ARC-AGI-2/data/evaluation/` (120 tasks). Task files are `{id}.json` with `{train: [{input, output}], test: [{input, output}]}`.
- **ARC runner** — `bun domains/arc-agi-2/src/runner.ts` outputs to `runs/<auto-name>/` by default. Each run produces `runs/<name>/run.json` (config + results + summary) and `runs/<name>/sessions/<taskId>_a<N>/agent-0.jsonl`. Key flags: `--runs-dir <path>` (default: `runs`), `--name <name>` (override auto-name), `--resume <run-name>` (skip completed tasks), `--count <n>`, `--task-id <id>`, `--model`, `--thinking`, `--num-attempts <n>` (default 2), `--max-agents <n>` (default 10), `--shard <n> --num-shards <n>`, `--stream`, `--no-log`. Old `--log-dir`, `--results-dir`, `--label` flags are removed. Defaults to `claude-opus-4-6` model and `high` thinking.
- **ARC eval results (Feb 28 2026)** — 79/120 (65.8%) on eval split with Opus + high + pass@2. Results: `pi-rlm/results/arc/2026-02-28T17-56-02-573Z-merged.json`. Arcgentica comparison: 85.3% (102.3/120). Gap analysis: `NEXT-gap-analysis.md`.
- **Sharded parallel runs** — `pi-rlm/run-eval-sharded.sh` launches 10 parallel workers. Merge with `bun src/merge-results.ts results/arc/*shard*.json`.
- **Test count: 160 tests (112 pi-rlm + 48 domain)** — pi-rlm includes 11 REPL coherence tests, 4 system prompt override tests, flat agent pool tests. Domain package has its own test suite at `domains/arc-agi-2/test/`.
- **Trace export CLI** — `bun pi-rlm/src/trace-export.ts` with two subcommands: `rollout` (session JSONL → sparse-v1.0 Rollout JSONL) and `sharegpt` (Rollout or sessions → ShareGPT JSONL for SFT). Supports `--filter correct|all|wrong`. Direct pipeline: `sharegpt --sessions <dir> --results <file> --out <file>`.
- **Domain package workspace dependency** — `domains/arc-agi-2` depends on `pi-rlm` via `"pi-rlm": "workspace:*"` with `main: "./dist/index.js"`. Changes to pi-rlm exports require a `tsc` build before domain typecheck passes.
- **Viewer API** — `bun domains/arc-agi-2/src/viewer.ts` serves on port 3335. Single endpoint `/api/runs` returns array of run.json objects (with `_activeSessions` annotation). Session traces via `/api/session/<runName>/<sessionDir>`. WebSocket at `/ws` with `subscribe`/`unsubscribe` messages using composite session IDs. FileWatcher watches `runs/` with three-level nesting. Configurable root via `VIEWER_RUNS_ROOT` env var. Includes `startPeriodicRescan()` for Modal volume compatibility (additive to `fs.watch()`). **Modal deployment:** `modal serve rl/modal_app.py` starts viewer at printed URL; run evals simultaneously with `modal run rl/modal_app.py::evaluate` — traces appear in ~10 seconds.
- **agent-runner.ts CLI** — `bun domains/arc-agi-2/src/agent-runner.ts` is the single-task CLI for the unified eval pipeline. Reads task JSON from stdin (`--task-from-stdin`), outputs JSON result to stdout, debug to stderr. Key flags: `--base-url`, `--model`, `--task-id`, `--max-turns`, `--timeout`, `--top-logprobs`, `--num-attempts`, `--max-agents`, `--run-dir`, `--session-id`, `--temperature`, `--top-p`, `--thinking`. Used by `eval_loop.py` as a subprocess.
- **rl/ eval pipeline (unified harness)** — `eval_loop.py` is now a thin subprocess dispatcher: takes `vllm_base_url`, spawns `bun run agent-runner.ts` per task with task JSON on stdin, collects JSON results. `modal_app.py::evaluate()` starts vLLM as HTTP server (`_start_vllm_server` + `_wait_for_server` polling `/health` every 2s), passes `vllm_base_url` to `evaluate_tasks()`. Python no longer imports vllm/transformers/ArcInteraction — Bun owns all agent logic. Run: `modal run rl/modal_app.py::evaluate --model Qwen/Qwen3-8B --dataset ARC-AGI-1 --count 5`. Download traces: `modal run rl/modal_app.py::pack_run --run-name <name>` then `modal volume get sdpo-arc-data eval-runs/<name>/run.tar.gz .`. 29 Python tests: `cd rl && uv run python -m pytest tests/ -v`.
- **8B/32K models hit context overflow within 3-4 turns on ARC** — System prompt (~3200 tokens) + task description + model generating ~3000 tokens/turn = overflow by turn 3. Fundamental constraint for 32K-context models on multi-turn ARC. Training must either: (a) use shorter prompts, (b) truncate old turns, (c) use models with longer context, or (d) reduce output verbosity.
