# Project Playbook

## Strategies and Hard Rules

- **Token depth = accuracy for agentic solvers** — On ARC-AGI-2, 3.5x more tokens/attempt → 20pp more accuracy (594k tokens at 65.8% vs 2M tokens at 85.3%). Same model, same reasoning level. When reproducing results, match token expenditure not just configuration flags. Check turn limits, timeouts, and action budgets — these gate exploration depth.
- **Explore unfamiliar data with python3 one-liners before coding the transformer** — For JSONL/JSON data transformation tasks, run 3-5 python3 one-liners to: count event types, print one example of each, trace join keys between files. Takes ~90 seconds, eliminates all guesswork.
- **Arcgentica's `success` field is self-eval, not ground truth** — Their attempt JSON `test_results[].success` reflects the agent's internal check, not whether the output matches the ARC answer key. Actual scoring requires running `score.py` against the solutions.

- **Edit existing files, Write only new ones** — For files that already exist, always use Read → Edit, even if the spec says to "update" them. Reserve Write for files that genuinely don't exist yet. The Write tool can error on existing files if they weren't Read first in the current session. Before creating any file, Glob for it first — prior sessions may have already created it.
- **Well-specified predictions eliminate design-time** — When the PROMPT includes concrete implementation approaches (not just "implement X" but "use Object.defineProperty with getters on a closure"), Ralph executes without deliberation. Invest time in prediction specificity. For 5+ file builds, specify every function signature (name, parameters, return type, defaults) — the arc session created 7 files in 8.5 minutes because nothing was left to deliberate.
- **Port reference implementation prompts, not just code** — When adapting an existing system (like arcgentica → pi-rlm), study the system prompts first. Battle-tested prompts encode methodology that transfers across languages. The code is secondary — the prompt is the product.
- **Check git status exactly once at session start** — Run `git rev-parse --git-dir 2>/dev/null` early. If it fails, skip all git operations for the rest of the session. Don't retry.
- **Batch typecheck+test is fine for TypeScript event-plumbing work** — When changes are purely additive (new fields on objects, new code paths guarded by flags), running `just typecheck && just test` once at the end catches everything. No need to run after each file edit.
- **Verify code state by grep, not by checkbox** — At session start, grep the actual source for key patterns (`submit\(`, `\bvar\b`, `snake_case_names`) rather than trusting `[x]` markers in the prompt file. Checkboxes from prior sessions may be stale, inverted, or wrong. Spend 30 seconds grepping; save 10+ minutes of wasted work.
- **When adding post-processing to IIFE-wrapped eval code, use `__retval` not `return`** — The eval runtime wraps code in `(async () => { ... })()`. If you need to append code after the user's statements (e.g., `__persist` calls), never use `return` for implicit returns — append `__retval = (expr)` instead, then `return __retval` at the very end. This ensures all post-processing code is reachable.
- **Stage specific files in commits, never `git add -A`** — `git add -A` in a repo with downloads/, symlinks, or prior session artifacts will capture hundreds of unrelated files. Always use `git add <specific-files>` or at minimum `git add pi-rlm/`. The arc-parity session's feature commit correctly staged specific files; the termination protocol's hardcoded `git add -A` pulled in everything.
- **Combine tasks that touch the same file** — When two tasks both modify the same file (like pass@K scoring + default model/thinking both in `arc-runner.ts`), implement them together. Avoids re-reading/re-editing and produces cleaner diffs.
- **Explore unfamiliar data with python3 one-liners before coding the transformer** — For JSONL/JSON data transformation tasks, run 3-5 python3 one-liners to: (1) count event types, (2) print one example of each type, (3) trace the join keys between files, (4) verify assumptions about field presence. This takes ~90 seconds and eliminates all guesswork from the implementation.
- **Test the full pipeline on real data, not just each stage** — For multi-stage pipelines (A→B→C), run the entire pipeline end-to-end before shipping. Stage-level validation (A→B looks correct, B→C looks correct) can miss information loss at boundaries. The trace-export session caught a missing `system_prompt` only when running session→rollout→sharegpt as a complete pipeline.

## Code Patterns

- **`createTaggedOnEvent()` returns `{ handler, setUserMessage }`** — This is the pattern for event tagging in pi-rlm. The handler is passed to `agent.subscribe()`, and `setUserMessage()` is called lazily in `run()`/`call()` when the user message becomes known. Don't try to pass userMessage at construction time.
- **Debug vs normal TUI rendering** — `handleEvent` dispatches to `handleNormalEvent` or `handleDebugEvent`. They share only the loader component and `formatToolResultRaw`. The debug path uses plain `Text` components (not `Markdown`) to avoid streaming flicker.
- **Event depth vs spawn depth** — `eventDepth` (UI nesting level) and `currentDepth` (max-depth recursion guard) are separate fields in `AgentHandleOptions`. They increment together but serve different purposes.
- **AgentEvent has 8 types, not 6** — The actual `AgentEvent` union includes `turn_start`, `turn_end`, and `tool_execution_update` in addition to the commonly referenced 6. The debug renderer handles the important ones and silently ignores the rest via a switch default.
- **Cross-cutting concerns use separate subscribers** — Don't inline usage tracking or logging into the event-tagging handler. Subscribe separately via `agent.subscribe()`. The tagged handler's job is decoration; other subscribers consume the decorated events. This keeps responsibilities clean and is easier to test independently.
- **ARC adapter is a pure scope provider** — After the idiomatic refactor, the ARC adapter no longer extends `DomainAdapter` with `getAnswer()`. The `orchestrator.run()` return value IS the answer. The adapter just provides scope and prompts. For new domains, follow this pattern: adapter provides scope, orchestrator handles completion.
- **`let`/`const`/`var`/`function` all persist in EvalRuntime** — `scanDeclarations()` handles all declaration types. All persist across eval calls via `__persist`. The scanner uses `computeBraceDepth()` to only capture top-level declarations (skips nested function bodies).
- **`resolve(value)`/`reject(reason)` for orchestrator completion** — The orchestrator uses `resolve(value)` to complete with a result and `reject(reason)` to abort. There is no longer a separate domain-level submission function. `orchestrator.run()` returns the resolved value directly; reject throws.
- **DomainAdapter.generateSystemPrompt bypass pattern** — `adapter.generateSystemPrompt?.() ?? generateSystemPrompt(adapter)` is a one-line opt-in for adapters to bypass orchestrator framing. No mode flags, no conditional logic. Other adapters get the default orchestrator prompt unchanged. Use this for any "domain X needs special handling" without polluting the generic path.
- **Shared mutable counter for flat resource pools** — `{ count: number }` object passed by reference through all `createSpawnAgent` levels gives a flat pool counter in ~10 lines. Works because JS objects are pass-by-reference. Simpler than depth tracking for "total N agents" limits (vs "N levels deep" limits).
- **SessionLogger `message_end` with `usage` = assistant turn** — The session logger echoes both assistant messages and user/tool-result messages as `message_end` events. The discriminator: assistant turns have a `usage` field (input/output token counts); echo messages don't. This is the key for reconstructing conversations from JSONL.
- **Content blocks use `toolCall` not `tool_use`** — pi-agent-core serializes tool calls as `{type: "toolCall", arguments: {code}}`, not the Anthropic API's `{type: "tool_use"}`. The trace converter handles both for robustness.

## Pitfalls

- **Use `git add <specific-files>` for commits** — Never `git add -A` since the workspace contains large downloads/ and other content excluded by `.gitignore`.
- **Spec says "Create file X" but it may already exist** — Always verify before writing. Existing implementations may have design improvements over the spec.
- **Manual testing requires a live API key** — `pi-rlm --debug` hits a real LLM API. There's no mock adapter for offline testing of the TUI rendering pipeline. Consider adding a `--dry-run` or replay mode.
- **Multiple `agent.subscribe()` calls are additive** — pi-agent-core allows multiple subscribers on the same agent. They receive the same event object reference. If subscriber A mutates the event (adding `_agentId`, `_depth`), subscriber B sees those mutations. Useful for cross-cutting concerns but subscriber ordering matters.
- **CLI flag parsing without a library is brittle** — Optional-argument flags like `--log [dir]` require index-based `process.argv` filtering to distinguish `--log ./logs` from `--log --debug`. Works for 2-3 flags but will break if more are added. If the flag count grows past 4, adopt `commander` or `yargs`.
- **Export `resetCounter()` for any module-level state** — Module-level counters (like `nextAgentId` in event-tagging) persist across test files. Without a reset function, test assertions on specific IDs become order-dependent and flaky. Always export a reset function alongside any module-level mutable state.
- **Use `process.cwd()` for test fixture paths in vitest** — Under symlinked repos, `__dirname` and `import.meta.url` resolve through the symlink to the real filesystem path, breaking relative paths to fixtures. `process.cwd()` works because vitest sets cwd to the project root (`pi-rlm/`). This is vitest-specific.
- **Reflector overhead can exceed 50% of core session cost** — For fast sessions (<10 min), the termination + reflector + curator pipeline ran 2-3 times redundantly in the arc session, costing ~$1.86 on top of a $3.11 core session. Consider gating reflector runs on session duration or cost threshold (e.g., skip if core session < $2).
- **Reflector agents may edit source code during analysis** — The arc session's third reflector run modified `arc-runner.ts` while reading the log. Constrain reflector agents to read-only tools + the delta file write. Surface source fixes as delta recommendations, don't apply directly.
- **`scanDeclarations` is scope-blind — use `computeBraceDepth()`** — The declaration scanner matches `let`/`const`/`var`/`function` with a regex. It will find declarations inside nested function bodies, arrow functions, and class methods. Always use `computeBraceDepth()` to filter to depth-0 declarations only. The old `var` scanner had this same bug but it was never caught because no tests used `var` inside function expressions.
- **Framework-level API renames touch more files than you expect** — When renaming something like `submit()` → `resolve()`, grep all of `src/` after the obvious renames. In the idiomatic session, `memories.ts`, `demo-adapter.ts`, and `agent-handle.ts` all had straggler references that would have been missed without a catch-all grep.
- **Predict signal-to-noise, not raw size** — When predicting whether a refactored prompt/config will be "shorter," the real metric is "no irrelevant content," not "fewer characters." A domain-specific prompt can be longer than a generic one and still be better because every byte is task-relevant. Frame predictions around content relevance, not length.
- **Keyword-matching for banned prompt content needs context** — ARC_PREMISE uses "delegate" in "do not delegate unless genuinely helpful" — this is solver-appropriate language telling the agent NOT to delegate, not orchestrator framing. Checking prompts for banned words requires understanding context, not just substring matching.
- **Intermediate formats must carry downstream metadata** — When format X feeds into format Y, X must carry everything Y needs — not just what X's own spec requires. The sparse-v1.0 Rollout spec doesn't include `system_prompt` at the top level (it's in Steps), but the ShareGPT exporter needs it. Check data flow through the full pipeline before finalizing schemas.

## Environment

- **Project structure** — Source is at `pi-rlm/src/`. Tests via `just test`, types via `just typecheck`.
- **pi-tui Text component** — `Text(text, paddingX, paddingY)` with `setText()` for updates. Use for streaming content in debug mode. `Markdown` component exists but causes flicker on rapid updates.
- **ARC-AGI-2 data** — `downloads/ARC-AGI-2/data/training/` (1000 tasks), `downloads/ARC-AGI-2/data/evaluation/` (120 tasks). Task files are `{id}.json` with `{train: [{input, output}], test: [{input, output}]}`.
- **ARC runner** — `bun pi-rlm/src/arc-runner.ts --count 5 --debug` for dev validation. Supports `--model`, `--thinking`, `--task-id`, `--split`, `--log`, `--num-attempts <n>` (default 2), `--max-agents <n>` (default 10), `--shard <n> --num-shards <n>` for parallel runs. Defaults to `claude-opus-4-6` model and `high` thinking. Auth: uses `ANTHROPIC_API_KEY` from `.env` if set, falls back to OAuth.
- **ARC eval results (Feb 28 2026)** — 79/120 (65.8%) on eval split with Opus + high + pass@2. Results: `pi-rlm/results/arc/2026-02-28T17-56-02-573Z-merged.json`. Arcgentica comparison: 85.3% (102.3/120). Gap analysis: `NEXT-gap-analysis.md`.
- **Trace export CLI** — `bun pi-rlm/src/trace-export.ts` with `rollout` and `sharegpt` subcommands. Converts session JSONL → sparse-v1.0 Rollout → ShareGPT for SFT. See `NEXT-sft-export.md`.
- **Sharded parallel runs** — `pi-rlm/run-eval-sharded.sh` launches 10 parallel workers. Merge with `bun src/merge-results.ts results/arc/*shard*.json`.
- **Test count: 158 tests across 12 files** — Includes 11 REPL coherence tests for `let`/`const` persistence, re-declaration, cross-eval access, iterative refinement, error recovery, and mixed declarations. Plus 4 tests for system prompt override and flat agent pool.
- **Trace export CLI** — `bun pi-rlm/src/trace-export.ts` with two subcommands: `rollout` (session JSONL → sparse-v1.0 Rollout JSONL) and `sharegpt` (Rollout or sessions → ShareGPT JSONL for SFT). Supports `--filter correct|all|wrong`. Direct pipeline: `sharegpt --sessions <dir> --results <file> --out <file>`.
