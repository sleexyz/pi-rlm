# Multi-Turn SDPO for ARC-AGI-2 — polo ralph session

## Context

**Goal:** Train Qwen3-8B to solve ARC-AGI-2 tasks using multi-turn SDPO, built on top of the existing verl + SDPO infrastructure from kirby.

**Why it matters:** We have a working SDPO pipeline on Modal (`~/projects/kirby/sdpo_modal.py`) that trains on LiveCodeBench. We have a working ARC evaluation harness (`domains/arc-agi-2/src/runner.ts`) that supports `--provider vllm --format block`. The gap is connecting them: plug the ARC environment into verl's SDPO training loop.

**Full spec:** `specs/multi-turn-sdpo.md` — read this first.

**Core design principle: ZERO EXECUTION SKEW between training and inference.** The model trained with SDPO will be evaluated using the existing TypeScript runner. Training must use the identical execution environment — same `EvalRuntime`, same `createArcAdapter()` scope, same variable persistence, same feedback formatting. We achieve this by reusing the pi-rlm TypeScript execution stack directly via a long-lived subprocess.

**What verl/SDPO already provides (DO NOT REBUILD):**
- Multi-turn rollout engine (`tool_agent_loop.py` — state machine)
- SDPO loss (top-K KL divergence, default K=100 with tail bucket, EMA teacher, IS correction)
- GRPO advantage estimation, rollout correction
- vLLM integration (rollouts, weight sync)
- Modal deployment (volumes, GPU config, image)
- WandB logging, checkpointing

**What pi-rlm/domains already provides (DO NOT REBUILD):**
- `EvalRuntime` (`pi-rlm/src/eval-runtime.ts`) — persistent JS execution context with declaration scanning, async IIFE wrapping, console capture, timeout. This is the execution kernel.
- `createArcAdapter()` (`domains/arc-agi-2/src/adapter.ts`) — scope setup with all grid helpers + task data. Single source of truth for what the model sees.
- `parseReplBlocks()` (`pi-rlm/src/repl-stream.ts`) — code-block parsing.
- `generateCodeBlockSystemPrompt()` (`pi-rlm/src/repl-stream.ts`) — system prompt for code-block format.
- `toolResultToUserMessage()` format (`pi-rlm/src/repl-stream.ts`) — feedback format: `"Code executed:\n```js\n{code}\n```\n\nREPL output:\n{output}"`.
- Grid helpers (`domains/arc-agi-2/src/grid-helpers.ts`) — all 20+ functions.
- ARC prompts (`domains/arc-agi-2/src/prompt.ts`) — ARC_PREMISE, ARC_REFERENCE.

**What we need to build:**
1. `sandbox-server.ts` — long-lived TS process that wraps `EvalRuntime` + `createArcAdapter()`. Reads JSON-line commands from stdin, executes code, writes results to stdout. ~50 lines. Lives in `domains/arc-agi-2/src/`.

3. `arc_interaction.py` — implements verl's `Interaction` interface. Parses code blocks, delegates to `js_sandbox.py`, formats feedback, computes reward. ~100 lines.
4. `parser.py` — port of `parseReplBlocks` to Python. 10-line regex.
5. `arc_data.py` — ARC tasks to verl parquet format. System prompt generated by calling TS at data-prep time (not ported to Python).
6. Config — `sdpo_arc.yaml` adapted from kirby's SDPO config.
7. Modal deployment — adapted from `sdpo_modal.py`. Image includes Bun + TS source files.

### Phase 1: Sandbox Server + Python Wrapper

The foundation — everything depends on correct code execution with zero skew from pi-rlm.

#### 1a. Sandbox Server (`domains/arc-agi-2/src/sandbox-server.ts`)

A long-lived process that wraps `EvalRuntime` + `createArcAdapter()`. JSON-line protocol over stdin/stdout.

- [ ] Import `EvalRuntime` from `pi-rlm/src/eval-runtime.ts`
- [ ] Import `createArcAdapter` from `./adapter.ts`
- [ ] On `{"type":"init","task":{...}}`: create adapter via `createArcAdapter(task)`, create `EvalRuntime` with adapter scope + `submit()` function
- [ ] On `{"type":"eval","code":"..."}`: call `runtime.eval(code)`, return `{"type":"result","stdout":"...","stderr":"...","returnValue":...,"error":...,"submitted":...}`
- [ ] `submit()` sets a flag + stores the value, included in the result JSON
- [ ] On `{"type":"shutdown"}`: exit cleanly
- [ ] Read from stdin line-by-line, write JSON-line responses to stdout

#### 1b. Python Sandbox Wrapper (`rl/js_sandbox.py`)

Manages long-lived `bun` subprocesses — one per rollout instance.

- [ ] `class SandboxSession`: wraps a single `bun` subprocess
  - `__init__(task: dict)`: starts `bun run domains/arc-agi-2/src/sandbox-server.ts`, sends `init` command with task
  - `eval(code: str) -> dict`: sends `eval` command, reads result JSON
  - `close()`: sends `shutdown`, kills process
- [ ] `class SandboxPool`: manages multiple sessions by instance_id
  - `get_or_create(instance_id: str, task: dict) -> SandboxSession`
  - `close_all()`
- [ ] Timeout handling: if `eval` doesn't return within N seconds, kill and restart subprocess
- [ ] Error handling: if subprocess dies, return error result

### Phase 3: ARC Interaction

The bridge between verl's rollout engine and the TS execution environment.

#### 3a. Read verl's Interaction Interface
- [ ] Read `~/projects/kirby/downloads/SDPO/verl/interactions/base.py` — understand the interface contract

#### 3b. ArcInteraction (`rl/arc_interaction.py`)
- [ ] Subclass verl's `Interaction`
- [ ] Owns a `SandboxPool` for managing per-instance TS subprocesses
- [ ] `generate_response(instance_id, messages, **kwargs)`:
  1. Extract latest assistant message
  2. Parse ```js code blocks via `parse_code_blocks()`
  3. Execute each code block via `SandboxPool.get_or_create(instance_id, task).eval(code)`
  4. Compute reward: 1.0 if submitted and accuracy == 1.0, else 0.0
  5. Format feedback matching `toolResultToUserMessage`: `"Code executed:\n```js\n{code}\n```\n\nREPL output:\n{output or '(no output)'}"`
  6. should_terminate = submitted or turn >= max_turns
  7. Return `(should_terminate, feedback, reward, {"accuracy": reward, "submitted": submitted is not None})`
- [ ] Handle edge cases: no code blocks in response, sandbox errors, timeout
- [ ] Cleanup: close sandbox sessions when rollout completes

## Instructions

### General
- Python RL code goes in `rl/` at the project root
- New TS code (sandbox-server.ts, generate-prompt.ts) goes in `domains/arc-agi-2/src/` — it's domain code
- Python 3.11+
- **Do NOT rebuild what verl already provides** — no custom SDPO loss, no custom training loop, no custom vLLM integration
- **Do NOT reimplement what pi-rlm already provides** — no custom EvalRuntime, no porting grid helpers, no porting system prompts. Reuse via subprocess.
- Follow verl's patterns for the Interaction class — read the GSM8K example carefully

### Key Design Decisions
- **Zero execution skew** — the sandbox-server.ts uses the same `EvalRuntime` + `createArcAdapter()` as the inference runner. Any change to scope, execution semantics, or grid helpers automatically applies to both training and inference.
- **Long-lived subprocess per rollout** — EvalRuntime has persistent context (variables survive across eval calls). The subprocess stays alive for the duration of a multi-turn rollout.
- **JSON-line protocol** — stdin/stdout, one JSON object per line. Simple, no HTTP overhead.
- **System prompt from TS** — generated at data-prep time by calling `generateCodeBlockSystemPrompt()` via bun, not ported to Python.
- **Feedback format from pi-rlm** — matches `toolResultToUserMessage()`: `"Code executed:\n```js\n{code}\n```\n\nREPL output:\n{output}"`. Hard-coded in Python (simple string template) but verified against TS source.
- **Use verl framework** — top-K KL divergence (K=100, with tail bucket), not the weaker logprob-advantage variant
- **4xA100-80GB** — same as kirby's proven setup

### Testing Strategy
- **Unit test** sandbox server + Python wrapper with 9 cases including variable persistence (Phase 1)
- **Unit test** parser with 6 cases matching `parseReplBlocks` behavior (Phase 2)
- **Unit test** ArcInteraction with 6 cases including multi-turn persistence (Phase 3)
- **Integration test** via Modal smoke test with 2 training steps (Phase 5b)
- **End-to-end test** via TypeScript runner against trained model (Phase 6b)

## If Stuck

1. **EvalRuntime import fails in sandbox-server.ts**: Check that `pi-rlm/src/eval-runtime.ts` can be imported with relative paths from `domains/arc-agi-2/src/`. May need tsconfig or path adjustment. Bun resolves TS imports natively.
2. **Subprocess communication issues**: JSON-line protocol is simple — one `JSON.parse(line)` per message. If stuck, test sandbox-server.ts standalone: `echo '{"type":"init","task":{...}}' | bun run domains/arc-agi-2/src/sandbox-server.ts`.
3. **verl Interaction interface unclear**: Read `gsm8k_interaction.py` — it's <100 lines and shows the full contract. Also read `tool_agent_loop.py` to see how interactions are called.
4. **Bun not on Modal**: `curl -fsSL https://bun.sh/install | bash` in image build. Verify with `bun --version`. Fallback: `node` (EvalRuntime uses `node:vm` which both support).
5. **verl data format unclear**: Read `~/projects/kirby/downloads/SDPO/data/preprocess.py` to see how LCB data becomes parquet. Match that format for ARC.
6. **Config doesn't load**: Start from a working config (`sdpo.yaml` + kirby's `modal_user.yaml`), change one thing at a time.
7. **Training diverges**: Copy kirby's exact hyperparameters first, only adjust batch size and rollout count for multi-turn.
8. If truly stuck: `<promise>BLOCKED</promise>`
